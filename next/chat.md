# Chat MVP (completed April 2026)

*A CLI prompt/response loop that talks to the CogServer via telnet. The
"inference" at this stage is trivial; the point is to make versus
interactive and to wire the integration seam end-to-end before the
walker adds real complexity.*

**Status: shipped.** The chat.py script at repo root, plus the
`inference.scm` generated by the atomspace module, implement this
design. Document kept here as a reference for how the MVP is wired;
the "Out of scope" and "Success criteria" sections describe what was
(and was not) built. Walker is the next build target; see
[walker.md](./walker.md).

## Scope for this build

Build **only the CLI chat interface plus a trivial inference procedure**:

- A `versus-respond` Scheme procedure: given a prompt string, finds
  FragmentAtoms whose surface text shares words with the prompt, returns
  one with provenance.
- A `versus-teach` Scheme procedure: given a sentence, registers it as
  a FragmentAtom and increments its `count` property.
- A Python CLI loop script that:
  - Connects to CogServer's telnet port (17001)
  - Enters Scheme REPL mode
  - Loops: read user input -> send Scheme call -> print response
  - Supports `:teach <sentence>` and `:quit` commands
  - Treats anything else as a prompt
- HCL wiring so `terraform apply` with `states = ["core", "chat-cli"]`
  produces everything, and the generated script is executable at
  `.versus/chat-cli.py`.

Explicitly out of scope for this build:

- Real inference (walker module)
- HTTP endpoint (deferred: reuses same `versus-respond` with a different
  transport)
- Wake/sleep cycles, learning
- Corpus ingest
- Conversational memory across turns
- Any kind of NLG beyond "pick a matching fragment and print its surface text"

## Success criteria

From a clean `terraform destroy`:

1. `terraform apply -var='states=["core","chat-cli"]'` succeeds with no errors.
2. CogServer container is up with telnet exposed on 17001.
3. Generated `.versus/chat-cli.py` is executable.
4. The following interaction works end to end:

```
$ python3 .versus/chat-cli.py
versus> connected to cogserver at localhost:17001
you> tell me about weather
bot> <no match - teach me with :teach first>
you> :teach weather is wet and cold
bot> learned: weather is wet and cold (count=1.0)
you> :teach summer is hot and dry
bot> learned: summer is hot and dry (count=1.0)
you> tell me about weather
bot> weather is wet and cold [provenance: fragment frag:weather is wet and cold, count=1.0]
you> :quit
versus> bye.
```

5. Fragment atoms persist across prompts within a session (in-RAM on CogServer).

## Module structure (as built)

The chat client is **not** a Terraform module. It is a static Python
script at repo root (`chat.py`) that reads connection details from
environment variables and has sensible defaults matching the state
fragment.

The earlier design (a `chat/` module generating `.versus/chat-cli.py`
via templatefile) was abandoned because templating Python feels wrong:
Python code does not benefit from state-fragment composition, and it
does not belong in `.versus/` alongside machine-consumed engine
artifacts. Environment variables are the clean boundary between
operator-facing tools and infrastructure-declared config.

Server-side Scheme (inference.scm.tftpl) lives in the `atomspace/`
module. This is a deliberate design choice: the inference procedures
run inside the CogServer (atomspace-side), and the atomspace module
is what controls what Scheme the CogServer loads at startup.

## Configuration (as built)

Via env vars or CLI args to `chat.py`:

- `COGSERVER_HOST` (default: `localhost`) or `--host`
- `COGSERVER_TELNET_PORT` (default: `17001`) or `--port`

No state fragment. The atomspace state fragment's `cogserver.telnet_port`
is the canonical value; `chat.py` matches the same default. If an
operator changes the port in `states/core.yaml`, they set the matching
env var when running `chat.py`.

## Atomspace-module changes (companion to this build)

Two changes land in the atomspace module as part of this work:

### 1. Expose telnet port 17001

Currently only 18080 (HTTP/WebSocket) is published. Add 17001 (telnet)
to the docker-compose template. No state-fragment change required; the
telnet port is part of the atomspace substrate, not something the
operator tunes.

### 2. Generate `inference.scm` and load it at startup

A third template renders `inference.scm` with the `versus-teach` and
`versus-respond` procedures. The compose command's startup loads it
alongside atom-schema.scm and decay-rules.scm.

The null_resource cogserver triggers are extended to include
inference.scm's hash, so changes to the inference template recreate
the container cleanly.

## Server-side Scheme (inference.scm)

Lives at `atomspace/templates/inference.scm.tftpl`. Pseudo-spec:

```scheme
(use-modules (opencog) (ice-9 format))

(define-public (versus-teach sentence)
  ;; Register a FragmentAtom keyed by "frag:<sentence>".
  ;; Initialize its count property if new; increment if existing.
  ;; Return a human-readable status string.
  ...)

(define (word-in-fragment? word fragment-name)
  ;; True if `word` appears as a whitespace-bounded token in
  ;; the substring of `fragment-name` after "frag:".
  ...)

(define-public (versus-respond prompt)
  ;; Split prompt on whitespace.
  ;; Find all Concept atoms whose name starts with "frag:" and whose
  ;; surface text contains at least one prompt word.
  ;; If none, return an explanatory string.
  ;; Otherwise, pick the highest-count match, return its surface text
  ;; plus a provenance annotation ("[provenance: fragment ..., count=...]").
  ...)
```

The exact Scheme is rendered from the template; there are currently no
template parameters because the inference is fully static at this stage.
When the walker lands, `versus-respond` is *replaced* (not augmented)
with a real inference walk.

## Python CLI shape (chat-cli.py.tftpl)

A single Python 3 file, stdlib-only (socket, sys, time). Two templated
values: `${cogserver_host}` and `${cogserver_telnet_port}`. Everything
else is static Python.

Core loop:

```python
s = connect_and_enter_scheme_mode(HOST, PORT)
while True:
    line = input("you> ").strip()
    if line == ":quit":
        break
    if line.startswith(":teach "):
        expr = f'(versus-teach "{escape(line[7:])}")'
    else:
        expr = f'(versus-respond "{escape(line)}")'
    response = send_and_read(s, expr)
    print(f"bot> {clean(response)}")
```

The `connect_and_enter_scheme_mode` helper:
1. Opens a TCP socket to the telnet port.
2. Drains the greeting banner.
3. Sends `scheme\n` to enter Scheme mode.
4. Drains the mode-switch confirmation.

The `send_and_read` helper:
1. Sends the Scheme expression with a trailing newline.
2. Reads until the next REPL prompt appears (heuristic: `"scheme@"` or
   `"guile>"` in the received buffer).
3. Returns everything before that prompt.

The `clean` helper strips echoed commands, trailing prompts, and
excessive whitespace from the raw response.

## Why CLI via telnet (and not HTTP, WebSocket, docker exec)

- **Telnet is simplest and persistent.** CogServer's telnet interface
  is the canonical OpenCog operator interface, designed for this use.
  Scheme-mode commands are line-oriented and stay in the same process,
  so atomspace state (taught fragments) persists across prompts.
- **HTTP** (port 18080) would require wrapping each prompt in an HTTP
  request, and CogServer's HTTP interface is less natural for Scheme
  evaluation. HTTP chat mode is on the roadmap but not the MVP.
- **WebSocket** (also 18080) works but is async-by-default and adds
  protocol complexity. Save for the HTTP-mode chat later.
- **`docker exec`** spawns a fresh Guile per prompt, losing atomspace
  state between calls. Not acceptable for a stateful chat.

## Open questions to settle during the build

- **Greeting drain timing.** CogServer's telnet greeting may be larger
  or faster than expected. A short timeout plus a retry loop is safer
  than a fixed wait; see chat-cli.py.
- **REPL prompt detection.** CogServer's Scheme-mode prompt string
  needs empirical verification. Current guess: `"scheme@"` per typical
  Guile REPL behavior. If different, the prompt-detection heuristic
  needs adjustment.
- **Response parsing.** Scheme-mode output may interleave prompts,
  expression echoes, and result values. The `clean` helper is heuristic;
  if responses look noisy, we iterate.
- **Fragment lookup via `cog-get-atoms`.** The `versus-respond`
  implementation iterates over all Concept atoms and filters by name
  prefix. This is O(N) per prompt but fine for POC scale. When atom
  counts grow, switch to type-indexed lookup or a dedicated
  FragmentAtom type.

## What this does NOT yet demonstrate

This MVP is deliberately shallow. What it does NOT prove:

- That versus can learn from a corpus (no wake/sleep cycles yet).
- That versus's generation is anything beyond concatenation of taught
  sentences (no walker mechanism, no fragment stitching).
- That fragment-granularity design choices are correct (all fragments
  here are whole-sentence-scale because `:teach` treats the whole
  sentence as one fragment).
- That provenance traceability generalizes (it is almost trivial at
  this scale: one fragment, one source sentence, direct mapping).

What it DOES prove:

- The integration seam works: state fragment -> Terraform -> generated
  Scheme -> loaded in CogServer -> callable from host via telnet.
- Provenance is exposed structurally in the response.
- The HCL surface can produce both server-side Scheme and client-side
  Python from the same state fragment family.
- Future modules (walker, ingest) can replace `versus-respond`'s
  internals without changing the chat module or the client script.
